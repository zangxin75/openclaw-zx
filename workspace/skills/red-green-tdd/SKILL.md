# Red/Green TDD - AI编程代码质量保障

基于 Django 创始人 Simon Willison 的 Red/Green TDD 方法论，为 AI 编程时代提供代码质量保障框架。

## 核心理念

> **不要只告诉 AI "做什么"，更要明确告诉 AI "做成什么样算对"**

当 AI 能替代大部分代码编写工作时，开发者的核心竞争力是**定义"什么是正确的代码"**的能力。

## Red/Green TDD 三步法

```
🔴 Red    → 先写测试，定义需求
🟡 Verify  → 运行测试，确认失败  
🟢 Green   → AI写代码，测试通过
```

### 第一步：先写测试（Red）

**原则**：不要先写功能代码，先让 AI 生成测试用例

**操作**：
- 向 AI 明确开发需求
- 让 AI 生成对应的测试用例
- 核心是定义"要实现什么"

**示例**：
```
用户：写一个"输入文章，输出所有标题"的函数

✅ 正确做法：
"先为这个函数写测试用例：
 1. 输入空字符串，返回空列表
 2. 输入带一个标题的markdown，返回该标题列表
 3. 输入带多个标题的markdown，返回所有标题列表
"

❌ 错误做法：
"帮我写一个提取标题的函数"
```

### 第二步：确认红灯（Verify）

**原则**：必须运行测试确认失败，验证测试有效性

**为什么**：
- AI 可能生成"永远通过"的无效测试
- 只有测试失败，才能证明它能检验功能

**操作**：
```bash
# 运行测试
pytest test_extract_titles.py

# 期望结果：FAILED (测试失败是预期行为)
```

### 第三步：实现功能（Green）

**原则**：让 AI 编写功能代码，直到测试通过

**操作**：
1. 让 AI 编写功能代码
2. 运行测试
3. 🟢 通过 → 继续下一步开发
4. 🔴 失败 → 让 AI 重写，直到通过

## 配套技巧：First Run Tests

**每次开启新任务时的第一句话**：
```
"先运行测试套件，确保当前代码状态正常"
```

**三大好处**：
1. **防止破坏** - AI 知道有测试，修改时会主动考虑运行测试
2. **规模感知** - AI 通过测试数量判断项目规模，编写更谨慎
3. **主动测试** - AI 感知到重视测试后，会主动为新需求写测试用例

## 实践 Checklist

### 开始新功能前
- [ ] 先让 AI 写测试用例
- [ ] 运行测试，确认失败（Red）
- [ ] 确认测试覆盖了边界情况

### 实现功能时  
- [ ] 让 AI 编写功能代码
- [ ] 运行测试，直到通过（Green）
- [ ] 测试通过后，再进行下一步开发

### 修改现有代码时
- [ ] 第一句："先跑测试，确保当前状态正常"
- [ ] 修改后再次运行测试
- [ ] 如有测试失败，先修复再提交

### 代码审查时
- [ ] 检查是否有对应的测试用例
- [ ] 测试是否覆盖主要场景
- [ ] 运行测试，确保全部通过

## 适用场景

| 场景 | 推荐程度 | 说明 |
|------|----------|------|
| 新功能开发 | ⭐⭐⭐⭐⭐ | 必用，定义验收标准 |
| Bug 修复 | ⭐⭐⭐⭐⭐ | 先写复现测试，再修复 |
| 代码重构 | ⭐⭐⭐⭐⭐ | 确保重构不破坏功能 |
| 简单脚本 | ⭐⭐⭐ | 根据复杂度决定 |
| 探索性编程 | ⭐⭐ | 快速原型后可补充测试 |

## 快速开始模板

### Python 项目

```python
# test_feature.py
import pytest
from my_feature import extract_titles

def test_extract_titles_empty():
    """空输入返回空列表"""
    assert extract_titles("") == []

def test_extract_titles_single():
    """单个标题"""
    text = "# Hello World"
    assert extract_titles(text) == ["Hello World"]

def test_extract_titles_multiple():
    """多个标题"""
    text = "# Title 1\n## Title 2\n# Title 3"
    assert extract_titles(text) == ["Title 1", "Title 2", "Title 3"]
```

### JavaScript 项目

```javascript
// feature.test.js
const { extractTitles } = require('./feature');

describe('extractTitles', () => {
  test('空输入返回空数组', () => {
    expect(extractTitles('')).toEqual([]);
  });
  
  test('提取单个标题', () => {
    const text = '# Hello World';
    expect(extractTitles(text)).toEqual(['Hello World']);
  });
});
```

## 助手提醒规则

当用户请求编程任务时，我应该：

1. **检测编程任务** - 识别是否为代码编写/修改请求
2. **提醒 TDD** - 建议先写测试
3. **提供模板** - 给出测试用例框架
4. **监督执行** - 确保遵循 Red/Green 流程

### 提醒话术模板

**开始新功能时**：
> 建议采用 Red/Green TDD：
> 1. 先让我为你写测试用例定义需求
> 2. 运行测试确认失败（Red）
> 3. 然后实现功能直到测试通过（Green）
> 
> 这样可以确保代码质量，要我帮你写测试吗？

**修改代码时**：
> 提醒：先运行现有测试套件，确保当前状态正常，然后再进行修改。

**代码完成后**：
> 代码已编写，建议：
> 1. 运行测试验证功能正确性
> 2. 检查测试覆盖率
> 3. 通过后可以继续下一步

## 核心价值

1. **质量兜底** - 测试是 AI 代码的"及格线"
2. **需求明确** - 测试定义了"什么是正确的"
3. **防止破坏** - 修改代码时保证原有功能正常
4. **效率提升** - 减少返工，一次做对
